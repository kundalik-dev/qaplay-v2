Interview Questions

Curated QA interview questions with model answers. Click to expand.

Start by installing Playwright: npm init playwright@latest. Set up a folder structure with tests/, pages/ (Page Objects), fixtures/, and utils/. Configure playwright.config.ts with base URL, multiple browsers, retries, and screenshot-on-failure. Use test.extend() for custom fixtures that handle auth state. Store credentials in .env and load via dotenv. Finally wire up a GitHub Actions workflow to run tests on every PR.
Playwright recommends user-facing locators in priority order: getByRole() → getByLabel() → getByPlaceholder() → getByText() → getByTestId(). I prefer getByRole because it reflects accessibility and is resilient to style changes. I avoid CSS selectors tied to class names that change frequently, and use data-testid attributes as a fallback when the element has no semantic role.
Playwright's APIRequestContext lets you make HTTP requests in the same test. I use API calls for test setup (creating users, seeding data) and teardown rather than navigating through UI flows — this makes tests faster and less flaky. For asserting API responses directly I use expect(response).toBeOK() and check response.json(). I also use page.route() to mock specific endpoints when testing edge cases like 500 errors or slow responses.
My process: (1) reproduce reliably by running with --repeat-each=5. (2) Check Playwright traces (trace: 'on-first-retry') to see exactly what happened. (3) Common causes: race conditions fixed by replacing waitForTimeout with waitForSelector or waitForLoadState; shared state fixed by test isolation (separate storage state per test); environment issues addressed by consistent test data setup via API. I never use hard sleep() — always wait for observable application state.
I use GitHub Actions with a workflow that triggers on pull_request and push to main. The job runs Playwright with --reporter=github,html. HTML report artifacts are uploaded with actions/upload-artifact so anyone can download and view them. For speed I run tests in parallel with sharding: --shard=1/4 across 4 matrix jobs. Failing PRs block merge via branch protection rules. I also run a nightly full regression against the staging environment.
The test pyramid has three layers: unit tests (many, fast, cheap) → integration/API tests (medium) → UI/E2E tests (few, slow, expensive). In practice I use E2E tests only for critical user journeys (login → checkout, CRUD flows), API tests for business logic and edge cases, and rely on unit tests from devs for individual functions. This keeps the test suite fast. A common anti-pattern I avoid is an "ice-cream cone" where E2E tests dominate, causing slow pipelines and high flakiness.
I skip automation for: (1) exploratory testing — human intuition finds unexpected issues better. (2) One-time tests that won't run again. (3) Rapidly changing UI where maintenance cost exceeds value. (4) Tests that need visual/UX judgment — screenshot tools help but aren't perfect. (5) Tests that are inherently flaky due to third-party dependencies (payment gateways, email). The ROI formula is: (frequency × time saved per run) > (cost to automate + maintenance cost).
