Alerts / Dialogs — Playwright Practice

Hard Scenarios

Two very different worlds: native JS dialogs (alert/confirm/prompt/beforeunload — handled with page.on('dialog'), NOT locators) vs in-DOM dialogs (HTML <dialog>, custom modals, toasts — handled with normal locators).

1. Native alert() Beginner

Auto-dismissed by Playwright by default. To assert text, register a handler BEFORE the click: page.on('dialog', d => { expect(d.message()).toBe('...'); d.accept(); })

Interview note: a native dialog blocks the page. dialog.type() returns 'alert'; only accept() is valid (no text).

2. confirm() — OK / Cancel Beginner

Two branches to automate. d.accept() → returns true (OK), d.dismiss() → returns false (Cancel). Assert the resulting page state.

Best-practice assertion: don't just read dialog.message() — verify the side effect in #confirm-result after accept vs dismiss.

3. prompt() — text input Intermediate

Pass text into the dialog: d.accept('My answer'). Dismiss/empty → app uses default or null. Note the default value via d.defaultValue().

Greeting will appear here…

Tricky cases to cover: (a) accept with text, (b) accept with empty string, (c) dismiss → returns null.

4. beforeunload — leave-page warning Advanced

Fires on navigation/close when there are unsaved changes. In Playwright the dialog only appears if you trigger an actual unload; handle it via page.on('dialog') with d.type() === 'beforeunload', or pass runBeforeUnload to page.close().

State: clean (no warning)

5. Native HTML <dialog> element Intermediate

This is real DOM — NOT a JS dialog. Use normal locators: getByRole('dialog'), getByRole('button', {name:'Confirm'}). showModal() adds a backdrop & traps focus; show() is non-modal.

6. Custom Modal (overlay div) Intermediate → Advanced

The most common real-world case (React/Vue modals). It's just a styled div with role="dialog". Locate by role/testid; wait for it with expect(modal).toBeVisible(). Closing via overlay-click / Esc / X are 3 paths to test.

7. Toast / Snackbar (auto-dismiss) Advanced

Appears then disappears after ~3s — a classic flaky-test trap. Assert with web-first expect(toast).toBeVisible() immediately, then expect(toast).toBeHidden(). Avoid hard waitForTimeout.

8. Dialog Handling Cheat Sheet

Type Is it real DOM? How to handle in Playwright Key API
alert() No (browser-native) page.on('dialog', d => d.accept()) before click dialog.message(), accept()
confirm() No Register handler, then accept() (OK) or dismiss() (Cancel) accept() / dismiss()
prompt() No Pass text: d.accept('value') defaultValue(), accept(text)
beforeunload No Check d.type()==='beforeunload'; page.close({runBeforeUnload:true}) dialog.type()
HTML <dialog> Yes Normal locators; assert visibility & returnValue getByRole('dialog')
Custom modal (div) Yes Locate by role/testid; web-first toBeVisible() expect(loc).toBeVisible()
Toast / snackbar Yes (transient) Assert visible, then hidden; no hard sleeps toBeVisible() / toBeHidden()

⭐ Golden rule: register page.on('dialog', …) before the action that triggers a native dialog — otherwise it auto-dismisses and your handler never runs.
