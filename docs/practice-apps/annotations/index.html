Annotations — Playwright & Selenium Practice

Each section demonstrates why you'd use a specific annotation and what UI to interact with. Playwright: test.skip() · test.fixme() · test.slow() · test.step() · test.fail(). Selenium/TestNG: @Test(enabled=false) · @DataProvider · @Test(timeOut=) · @Test(groups={}).

1. Feature Flags & Environment Conditions Beginner test.skip(condition, 'reason')

Real-world: skip a test when a feature is behind a flag, or when running in production. Playwright: test.skip(!betaEnabled, 'Beta feature not active'). TestNG: @Test(enabled=false) or throw new SkipException("reason"). JUnit5: @Disabled("reason") · @DisabledIfSystemProperty.

2. Slow Operations — Async Data Loading Beginner test.slow()

Real-world: reports, file exports, heavy API calls exceed the default 30s timeout. Playwright: test.slow() triples timeout (30s → 90s). No condition needed — just call it. TestNG: @Test(timeOut = 10000). JUnit5: @Timeout(value=10, unit=SECONDS).

3. Known Bugs & Expected Failures Intermediate test.fixme() test.fail()

test.fixme('BUG-123') — test is skipped but flagged in report so the team tracks the broken feature. test.fail() — test is expected to fail; it passes only if the assertion actually fails (confirms bug still exists). TestNG: @Test(expectedExceptions=RuntimeException.class).

4. Multi-Step Flow — test.step() Practice Intermediate test.step('name', async () => …)

test.step() groups actions into named steps — visible in the HTML report & trace viewer. Makes failures easier to locate: "Failed in step: Fill shipping form" vs a cryptic line number. Allure for Selenium: @Step("Step name") on methods.

5. Data-Driven / Parameterized Tests Intermediate test.each(data) @DataProvider

Same test logic, multiple data sets. Playwright: for (const user of users) { test(user.role, async ({page}) => { … }) }. TestNG: @DataProvider(name="users") + @Test(dataProvider="users"). JUnit5: @ParameterizedTest + @MethodSource("userData").

6. Test Tags & Custom Annotations Advanced @smoke @regression @critical

Playwright: include tag in test name → test('@smoke login', …) → run with --grep @smoke. Or use test.info().annotations.push({type:'priority', description:'P0'}) for custom report data. TestNG: @Test(groups={"smoke","regression"}) + testng.xml filter. JUnit5: @Tag("smoke").

7. Playwright vs Selenium — Annotations Cheat Sheet

IntentPlaywright (JS)TestNG (Java)JUnit5 (Java)
Skip unconditionally test.skip() @Test(enabled=false) @Disabled("reason")
Skip conditionally test.skip(condition, 'reason') throw new SkipException("reason") @DisabledIfSystemProperty
Known broken (tracked) test.fixme('BUG-123') @Test(enabled=false) + Jira comment @Disabled("BUG-123")
Expected to fail test.fail() @Test(expectedExceptions=Exception.class) assertThrows()
Increase timeout (3×) test.slow() @Test(timeOut = 90000) @Timeout(value=90, unit=SECONDS)
Group / tag test('@smoke name', …) @Test(groups={"smoke"}) @Tag("smoke")
Data-driven for (const d of data) test(d.name, …) @DataProvider + @Test(dataProvider=) @ParameterizedTest + @MethodSource
Group steps (report) await test.step('name', async () => …) Allure @Step("name") Allure @Step
Custom annotation test.info().annotations.push({type, description}) Custom Java @interface Custom annotation + Extension
Attach file to report test.info().attach('name', {body, contentType}) Allure @Attachment Allure @Attachment
Run only this test.only(…) or describe.only(…) Run specific class/method via IDE/Maven @Test + run single method

⭐ Key Playwright distinction: test.fixme() keeps the test in the report as "skipped" so the team knows it's broken. Never use test.skip() for bugs — use test.fixme() so it stays visible.
