→Overview

Input fields are the foundation of form automation. The core scenarios you'll encounter in every real-world test suite are: typing text, clearing a field, reading the current value, checking disabled state, and asserting readonly behaviour.

Selenium uses sendKeys(), clear(), and getAttribute("value"). Playwright has fill() (which clears first), inputValue(), and built-in assertions like toHaveValue(). Cypress uses type(), clear(), and should("have.value").

1Type Text into an Input

The most common input action. In Selenium, sendKeys() appends to whatever is already in the field. Playwright's fill() clears first then types — use keyboard.type() to truly append. Cypress's type() appends like Selenium.

2Append Text and Press Tab

Playwright's fill() clears the field before typing, so to append you must use locator.click() to focus, then keyboard.press("End") to move the cursor, then keyboard.type() to append. Selenium's sendKeys() naturally appends. In Cypress, include {tab} in the type string.

3Read the Field Value

Getting the current text of an input requires reading the value property, not the element's visible text. In Selenium use getAttribute("value"). Playwright has a dedicated inputValue() method and the toHaveValue() assertion. Cypress chains should("have.value", "expected").

4Clear an Input Field

Always clear an input before typing new text to avoid appending to stale data. Selenium's clear() method empties the field and triggers the change event. Playwright uses fill("") — passing an empty string. Cypress has a dedicated clear() command.

5Disabled Input Field

A disabled input has the HTML disabled attribute. It is visible but cannot receive keyboard input or be focused. The automation goal is to assert this state — Playwright will throw if you attempt fill() on a disabled element due to actionability checks.

Disabled Input — Assert, don't type DISABLED

// Selenium Java

// Cypress
cy.get(sel)
  .should('be.disabled');

6Readonly Input Field

A readonly input can be focused and selected, but keyboard input does not change its value. Unlike disabled, isEnabled() returns true for readonly fields. Detect readonly by checking the readonly attribute via getAttribute("readonly"), which returns an empty string "" when present.

8FAQ

What's the difference between fill() and type() in Playwright? ⌄
fill() clears the input first, then sets the value atomically — it does not fire keyboard events. type() (via keyboard.type()) simulates real keystrokes and appends to the existing value. Use fill() for most form interactions; use keyboard.type() when you need to test key-by-key event handling or genuinely append text.
Why does Playwright fail when calling fill() on a disabled input? ⌄
Playwright runs actionability checks before interacting with an element: it confirms the element is visible, stable, and enabled. A disabled input fails the enabled check and the fill() call throws after the configured timeout. Assert the disabled state with toBeDisabled() instead of attempting to interact with it.
How do I check if an input is readonly vs disabled in Selenium? ⌄
Disabled: !element.isEnabled() returns true. The element cannot be focused or interacted with.
Readonly: element.isEnabled() returns true (it IS enabled!), but element.getAttribute("readonly") returns a non-null value. The key difference: readonly accepts focus and selection but not value changes.
How do I append text to an existing value in Playwright without clearing it? ⌄
Playwright's fill() always clears first. To truly append: (1) click the locator to focus, (2) press End to move cursor to end, (3) use page.keyboard.type("append text"). This simulates real keystroke events and preserves the existing value.
What selector strategy is best for input fields in automation? ⌄
Prefer data-testid attributes — they're stable across styling and layout changes. In Playwright: page.getByTestId("inp-movie-name"). In Selenium: By.cssSelector("[data-testid='inp-movie-name']"). For forms with clear semantic labels, Playwright's getByLabel("Movie Name") is even more human-readable and resilient.
How do I simulate pressing Enter after typing in an input? ⌄
Playwright: await locator.press("Enter") after fill(). Selenium: element.sendKeys(Keys.ENTER). Cypress: include {enter} in the type string — cy.get(sel).type("text{enter}"). This is commonly needed for search forms and login flows.
